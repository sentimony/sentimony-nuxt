# Організація аудитів і roadmap

Дата: 2026-07-22
Статус: погоджено

## Мета

Перетворити кореневі `AUDIT.md` і `ROADMAP.md` на масштабовану документаційну
структуру, у якій:

- завершені аудити накопичуються як окремі датовані документи;
- кожна активна або майбутня roadmap-ініціатива має власний файл;
- короткий roadmap не дублює специфікації та implementation plans;
- фактичний статус реалізації можна визначити з індексу без читання всіх документів;
- нові аудити й ідеї додаються без розростання одного кореневого Markdown-файлу.

## Ухвалений підхід

Використовувати стабільні тематичні файли та індекси:

- `docs/audits/` — append-only архів датованих аудитів;
- `docs/roadmap/` — живий product/engineering backlog, один файл на ініціативу;
- `docs/superpowers/specs/` і `docs/superpowers/plans/` — незмінне місце для
  детального дизайну й покрокових implementation plans.

Кореневі `AUDIT.md` і `ROADMAP.md` переносяться повністю. Файли-вказівники в
корені не залишаються. Внутрішні посилання оновлюються на нові шляхи.

## Архів аудитів

### `docs/audits/README.md`

Індекс описує призначення папки, naming convention
`YYYY-MM-DD-<topic>-audit.md` і список аудитів у зворотному хронологічному
порядку. Для кожного запису вказуються дата, тема та короткий результат.

Аудит після завершення вважається історичним знімком. Фактичні помилки можна
виправити в тому самому файлі, але зміни висновків після нової перевірки
оформлюються окремим аудитом або явно датованим доповненням.

### `2026-07-19-quality-audit.md`

Це повне перенесення поточного `AUDIT.md` без переписування історичного змісту.
Його твердження оцінюються станом на вказану в ньому дату, навіть якщо частину
знахідок уже виправлено пізніше.

### `2026-07-22-documentation-status-audit.md`

Новий аудит фіксує:

- статус усіх специфікацій і планів: реалізовано, частково або не реалізовано;
- доказ статусу через поточні файли, конфігурацію, тести та git history;
- розбіжність між порожніми plan checkboxes і фактичним кодом;
- покриття кожного пункту аудиту від 2026-07-19 у спеках і планах;
- знайдені неточності старого roadmap;
- поточний стан unit suite з окремим застереженням про незавершені локальні
  UI-зміни.

Аудит не змінює статуси в старих спеках і планах. Він є окремим джерелом
оцінки, а не прихованим переписуванням історичних документів.

## Roadmap

### `docs/roadmap/README.md`

Індекс є єдиною точкою входу в актуальний backlog. Ініціативи групуються за
статусом, а всередині групи — за пріоритетом:

1. `In progress`;
2. `Planned`;
3. `Ideas`;
4. `Completed`.

Для кожної ініціативи індекс містить посилання, пріоритет і одне речення про
результат. Дата `Last reviewed` показує свіжість саме індексу, а не окремих
історичних аудитів.

### Формат файла ініціативи

Кожен roadmap-файл має компактний спільний контракт:

```markdown
# Назва ініціативи

- Status: Idea | Planned | In progress | Blocked | Implemented
- Priority: P0 | P1 | P2 | P3 | Future
- Last reviewed: YYYY-MM-DD
- Related: посилання на аудит, spec і plan за наявності

## Навіщо
## Очікуваний результат
## Обсяг
## Залежності
## Критерії завершення
## Наступний крок
```

Roadmap-файл не містить покрокових команд, великих code snippets або детального
архітектурного дизайну. Для цього він посилається на відповідну spec/plan.

### Перенесення поточного roadmap

Активні пункти з кореневого `ROADMAP.md` розділяються на тематичні файли.
Застарілі твердження виправляються за поточним кодом і аудитом, зокрема:

- CI workflow і `typecheck` script уже існують, але Playwright і Netlify-preset
  gates ще не реалізовані;
- lazy media tabs уже реалізовані та переходять у `completed.md`;
- hidden artist detail exposure не можна вважати закритим;
- unit baseline не подається як зелений, доки поточний worktree має падіння;
- непокриті знахідки аудиту отримують окремі roadmap-ініціативи.

`completed.md` зберігає стислий історичний список закритих робіт і посилання на
реалізовані спеки/плани. Для завершених пунктів не створюються десятки нових
окремих файлів, якщо вони більше не потребують роботи.

## Нові майбутні ініціативи

### Cloudflare domain

`cloudflare-domain.md` починається як discovery і розділяє два незалежні рішення:

1. перенесення DNS/CDN/proxy до Cloudflare зі збереженням Netlify runtime;
2. окрема оцінка повного перенесення runtime на Cloudflare Pages/Workers.

Перша фаза не передбачає автоматичного рішення про другу.

### Sentry observability

`sentry-observability.md` охоплює client/server error tracking, source maps,
release/environment tagging, privacy filtering і правила alerting. Реалізація
починається лише після окремого дизайну обсягу даних і секретів.

### Власний UI і дизайн-система

`custom-ui.md` описує поступову заміну сторонніх presentation-компонентів на
Sentimony-owned компоненти без відмови від базових accessibility primitives.
`design-system.md` є залежністю: спершу фіксуються токени, стани, типографіка,
spacing і component contracts, потім UI-компоненти мігруються кластерно.

### Власний Swiper

`custom-swiper.md` є окремою дослідною ініціативою. Спочатку визначаються реальні
потреби поточного каталогу й performance baseline; рішення про власну реалізацію
не приймається лише заради видалення залежності.

### Візуалізація музичних хвиль

`audio-waveform.md` залежить від стабільного глобального player state. Discovery
має порівняти precomputed peaks із runtime Web Audio analysis, враховуючи SSR,
мобільну продуктивність, CORS/R2 і доступність seek-контролу.

### Подарунки Bandcamp-кодами

`bandcamp-code-gifts.md` описує безпечну одноразову видачу кодів користувачам.
Перед реалізацією потрібні правила eligibility, імпорт/зберігання кодів,
ідемпотентне claim API, аудит видачі та захист від повторного отримання. Ініціатива
залежить від profile/auth і mutation hardening.

## Зв'язок зі спеками й планами

- Існуючі спеки та плани не переміщуються й не перейменовуються.
- Roadmap-файли посилаються на них відносними Markdown-посиланнями.
- `PRODUCT.md` та інші згадки кореневого `ROADMAP.md` оновлюються на
  `docs/roadmap/README.md` або конкретну ініціативу.
- Майбутній roadmap item отримує spec лише перед дизайном реалізації, а plan —
  після погодження spec. Наявність roadmap-файла сама по собі не означає дозвіл
  на реалізацію.

## Обмеження

- Не змінювати production-код, дані, migrations або remote services.
- Не запускати `sync:*`, deploy або інші write-операції.
- Не редагувати наявні незавершені UI-зміни користувача.
- Не позначати фічу реалізованою лише через наявність spec/plan або порожніх
  plan checkboxes.
- Не дублювати повний текст аудитів, специфікацій чи планів у roadmap.

## Верифікація

Після перенесення:

1. `AUDIT.md` і `ROADMAP.md` відсутні в корені, а їхній зміст збережено в новій
   структурі.
2. `rg` не знаходить застарілих посилань на кореневі файли.
3. Кожен файл з `docs/roadmap/README.md` існує, а кожна ініціатива присутня в
   індексі рівно один раз.
4. Кожна spec/plan-пара має коректний статус у документаційному аудиті.
5. Пункти аудиту від 2026-07-19 або мають roadmap item, або явно позначені як
   уже виправлені з доказом.
6. `git diff --check` проходить.
7. У staged diff немає сторонніх змін користувача.
